<objective>
Integrate the narrow bridge flows into Streamlit so a verified `@ku.th` user can request access, a sponsor can approve from a login-gated email link, and approved users can reach the existing reservation UI without manual registration.
</objective>

<must_haves>
- New unapproved users can submit an access request with the chosen bridge fields.
- Sponsors can approve or deny only the requests addressed to them.
- Sponsor approval emails land in the app through a request-specific link and still require sponsor sign-in.
- Approved users can reach the existing reservation UI while pending users cannot.
</must_haves>

<threat_model>
- Unauthorized approval through link guessing: require sponsor login and compare the signed-in user ID or email against the request’s selected sponsor before showing decision controls.
- Improper self-upgrade to sponsor: never let the applicant choose `is_sponsor`; only bridge-seeded sponsor users can approve.
- Approval-state bypass into reservation tabs: gate all booking UI on the database-backed `approval_state` and `is_email_verified` checks, not on old session booleans alone.
- Approval spam: rate-limit sponsor email sends per access request so repeated reruns do not dispatch duplicate notifications.
</threat_model>

<objective>
Create the narrow backend foundation for the Streamlit bridge so the app can authenticate `@ku.th` users with one-time codes, persist sessions across refresh, and load authorization state from Neon instead of `st.secrets`.
</objective>

<must_haves>
- The app no longer depends on `st.secrets["credentials"]["usernames"]` to decide whether a user exists.
- Login codes are generated, stored, validated, and consumed through Neon-backed records.
- Browser refresh restores the authenticated user from a cookie-backed session token.
- The persisted user model uses independent flags instead of one exclusive role string.
</must_haves>

<threat_model>
- OTP replay: store only hashed codes, mark each code single-use, expire codes after a short TTL, and reject further use after `consumed_at` is set.
- Sponsor or admin forgery through cookies: store only an opaque session token in the cookie, hash the token in Neon, and re-read flags from the database on every fresh session.
- Shared-domain cookie leakage: never store approval state or roles directly in the cookie payload; use a unique cookie name and secure options where the component allows them.
- Secrets drift: move `DATABASE_URL`, `RESEND_API_KEY`, `RESEND_FROM_EMAIL`, `APP_BASE_URL`, `SESSION_COOKIE_NAME`, `SESSION_TTL_HOURS`, and `LOGIN_CODE_TTL_MINUTES` into runtime config and remove credential lookups from app auth.
</threat_model>

<objective>
Migrate existing manually registered users into the bridge identity store, preserve their current access after one `@ku.th` verification, and produce the minimal runbook and checks needed to operate the bridge safely.
</objective>

<must_haves>
- Existing secret-backed users can be imported into the bridge database with approved access preserved.
- Imported users must still verify their `@ku.th` email once before the bridge session is activated.
- The bridge has a repeatable sponsor-seeding and legacy-user bootstrap path.
- There is a concrete smoke-test checklist for operator use before deployment.
</must_haves>

<threat_model>
- Over-grant during migration: map only the known current role strings, fail closed on unknown role values, and print a migration summary for review.
- Name mismatch with reservation CSVs: preserve the old display name string exactly during import so reservation ownership still lines up.
- Re-running bootstrap scripts: make import and sponsor seeding idempotent by upserting on lowercase email.
- Hidden bridge drift: document required environment variables and smoke-test steps in the repo so the bridge can be redeployed without reconstructing knowledge from chat history.
</threat_model>
